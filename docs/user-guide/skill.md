# xAgent Skill 管理：创建、测试、发布与更新

## 适用对象

本文适合希望让 xAgent 稳定处理某类重复工作的用户和管理员。

## 这是什么

Skill 是可复用的工作方法。它把固定步骤、输出要求、注意事项和质量标准保存下来，让 xAgent 在处理同类任务时不必每次从头理解。

普通用户可以直接使用管理员准备好的公共 Skill，也可以在权限允许时创建或导入个人 Skill。大多数用户不需要从零配置，直接使用即可；熟悉后可以精调个人 Skill。管理员负责把稳定的个人 Skill 审核后沉淀为公共能力。

![xAgent Skill 管理页面，显示个人与公共 Skill 列表及操作入口](/img/manual/xagent-skill-management.png)

## 可见范围

xAgent 当前不对 Skill 做复杂的访问权限限制。公共 Skill 对所有用户可见，适合沉淀通用、稳定、可以被团队成员共同使用的工作方法。

如果某个 Skill 有特殊要求，例如只适合某个用户、包含个人工作习惯、面向特定客户流程，或不希望其他用户看到，请单独安装或保存到个人 Skill 库下。个人 Skill 只在当前用户范围内可见，其他用户看不到这个特殊要求的 Skill。

当前可以按下面方式理解 Skill 的状态和范围：

| 类型 | 谁能看到 | 能做什么 |
| --- | --- | --- |
| 草稿 Skill | 当前用户 | 编辑、测试，确认后发布到私人库 |
| 个人 Skill | 当前用户 | 使用、编辑、导出、删除；稳定后可以提交公共库审批 |
| 公共 Skill | 所有用户 | 查看、引用和使用；普通用户通常只读，管理员负责审核和维护 |

公共 Skill 不适合写入个人习惯、临时路径、私有账号、客户专属细节或不可公开的内部流程。有特殊要求的 Skill 应留在个人 Skill 库。

## 什么时候使用

下面场景适合使用技能：

- 任务经常重复。
- 每次都有固定步骤和格式。
- 需要稳定输出质量。
- 需要组合多个工具完成工作。
- 需要把个人经验沉淀给团队复用。

常见技能包括研究整理、报告生成、会议纪要、合同审查、数据分析、客服回复、投研分析、文档理解等。

如果想查看或参与修改面向实际任务场景的内置 Skill，可以阅读 [内置 Skill 文件](/docs/user-guide/builtin-skills)。docs 仓库中提供了一份 `skills/` 副本，方便大家通过 GitHub 提交建议和改进。

## 页面怎么看

Skill 管理 页面会展示技能卡片。常见信息包括：

| 信息 | 含义 |
| --- | --- |
| 技能名称 | 技能的显示名称或标识 |
| 描述 | 技能适合处理什么任务 |
| 可编辑 / 只读 | 当前用户是否可以修改 |
| 详情 | 查看技能说明、使用方式和配置 |
| 下载 / 发送 / 删除 | 针对可编辑技能的管理操作 |

普通用户主要看名称、描述和详情。不要只按名称选择技能，要看它是否适合当前任务。

## 基本用法

### 使用已有技能

1. 打开 Skill 管理。
2. 搜索任务关键词，例如“合同”“报告”“客服”。
3. 打开技能详情，确认用途是否匹配。
4. 回到 Agent会话，提交任务并说明目标和材料。
5. 如果需要使用某个技能，可以直接写“请按某某技能的方法处理”。

示例：

```text
请按“会议纪要”类方法整理我上传的录音转写，输出决策、待办、负责人和截止时间。
```

### 创建或更新个人技能

适合已经熟悉固定流程、希望精调个人能力的用户。创建或更新前先准备：

- 这个技能解决哪类任务。
- 用户需要提供哪些材料。
- 处理步骤是什么。
- 输出格式是什么。
- 哪些情况需要人工确认。
- 哪些内容不应该写入技能。

不要把临时文件路径、账号密码、token、验证码或一次性客户信息写进技能。

除了在 Skill 管理中创建或导入，也可以在主 Agent 会话中描述想要创建的 Skill。xAgent 会自动创建一个 Skill 构建会话，先了解需求，再生成草稿，并在你确认后完成发布。也可以直接通过智能体创建一个专门的 Skill 构建会话。

更新已有 Skill 时，可以在更新 Skill 的子会话里描述要更新哪个 Skill。xAgent 会自动将对应 Skill 拷贝一个副本到会话目录下，并以这份副本作为草稿，根据你的需求进行修改。确认后再发布更新。

### 发布到私人库

创建或编辑 Skill 时，系统会先生成草稿。草稿适合测试和调整，不建议直接当成团队标准能力使用。

当草稿内容确认可用后，在 Skill 管理中执行“发布私人库”。发布后，这个 Skill 会成为当前用户的正式个人 Skill。后续 Agent 会话可以引用它，当前用户也可以继续编辑、导出或删除。

### 提交公共库审批

个人 Skill 经过真实任务测试并确认稳定后，可以提交到公共库审批。提交时，系统会复制当前正式个人 Skill 的快照交给管理员审批；你后续继续编辑个人 Skill，不会影响已经提交的审批快照。

推荐流程是：

1. 先创建或导入 Skill，形成草稿。
2. 在自己的任务里测试，补充示例、边界和质量标准。
3. 发布到私人库，作为个人 Skill 使用。
4. 使用一段时间后，如果适合多人复用，再提交公共库审批。
5. 管理员审核通过后，进入公共库，所有用户可见。

管理员审核时应重点看用途是否通用、描述是否清楚、是否包含敏感信息、是否依赖私有账号或特定外部系统、是否适合所有用户看到。

### 让技能更好用

一个好技能通常具备：

| 要素 | 说明 |
| --- | --- |
| 明确用途 | 一句话说明适合什么任务 |
| 输入要求 | 用户需要提供哪些材料 |
| 操作步骤 | 按顺序说明怎么处理 |
| 输出标准 | 结果格式、字段、质量要求 |
| 风险提醒 | 什么时候要确认或停止 |
| 示例 | 给普通用户一个可复制的任务写法 |

如果技能越来越复杂，建议拆成多个小技能，而不是把所有场景塞进一个技能。

## 常见场景

### 固定报告

```text
请按“数据可视化报告生成器”的方式，基于 sales.csv 生成一份本周经营分析报告。
```

### 客服回复

```text
请按客服支持流程，根据客户问题生成回复草稿，先不要发送。
```

### 合同审查

```text
请按合同审查方法阅读这份合同，重点检查付款、交付、违约和争议解决条款。
```

### 生成新技能草稿

```text
请根据我下面这套固定工作方法，帮我整理成一个可维护的 Skill 草稿。
```

## 管理建议

- 公共 Skill 要面向多人，避免写个人习惯。
- Skill 描述要给使用者看，不要只写内部术语。
- 常用 Skill 应配清楚示例任务。
- 涉及外部发送、删除、账号授权的步骤，要写明先确认。
- 过期或不稳定 Skill 应标记清楚，避免普通用户误用。
- 提交公共库前，先在个人 Skill 中完成足够测试。
- 公共库审批通过前，不要承诺团队成员已经可以稳定使用。

## 常见问题

### 技能是否等同于工具？

不是。工具执行动作，技能沉淀方法。技能可以指导 xAgent 使用工具，但不替代工具权限。

### 技能越多越好吗？

不是。技能太多会让用户难以选择。应优先沉淀高频、稳定、有明确价值的流程。

### 普通用户必须选择技能吗？

不必须。管理员可以把技能关联到智能体，普通用户直接提交任务即可。

### 技能修改后是否立即生效？

草稿修改只影响当前草稿测试。发布到私人库后，后续任务可以使用新的个人 Skill。公共库 Skill 需要经过管理员审批和维护流程，已经运行中的任务是否采用新内容，以当前页面和会话行为为准。
