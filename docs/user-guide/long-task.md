# 长任务

> 状态：实验性能力，页面和字段可能变化。

## 适用对象

本文适合需要让 xAgent 处理多步骤、长时间或阶段性交付任务的用户。

## 这是什么

长任务是不能靠一次短回复稳定完成的任务。它可能需要读取大量材料、分阶段执行、等待确认、调用外部系统、生成多个文件，或在较长时间内持续跟进。

普通用户不需要知道内部如何拆分，只需要把目标、材料、阶段和验收要求说清楚。

## 什么时候使用

下面任务适合按长任务组织：

- 研究多个来源并生成报告。
- 分析大量文件或数据。
- 持续跟进客户、邮件或外部事件。
- 每天、每周重复生成固定产物。
- 需要先计划、再确认、再执行。
- 需要多个中间结果和最终交付物。

简单问答、短文本改写、单个文件小结通常不需要长任务方式。

复杂长任务中，主 Agent 可能会把部分工作拆给子 Agent。普通用户不需要手动管理子 Agent，只需要在原会话里查看阶段结果、补充材料和确认关键动作。

## 提交方式

建议长任务第一条消息包含：

| 内容 | 示例 |
| --- | --- |
| 最终目标 | 生成一份客户反馈分析报告 |
| 材料范围 | 使用我上传的 5 份访谈记录 |
| 阶段划分 | 先列计划，确认后再分析 |
| 中间结果 | 先输出主题分类和证据 |
| 最终交付 | 生成 HTML 报告 |
| 确认点 | 外发或删除前必须确认 |

示例：

```text
请基于我上传的 5 份访谈记录做用户需求分析。
先列处理计划，等我确认后再提取主题、证据和机会点。
最后生成一份 HTML 报告，并把原始证据保留在附录里。
```

## 推荐流程

### 先让 xAgent 列计划

复杂任务建议先写：

```text
请先列出处理计划，不要立即执行。
```

确认计划后，再让 xAgent 继续。

### 分阶段验收

长任务不要只看最终结果。建议在每个阶段检查：

- 材料是否读取正确。
- 分类是否合理。
- 证据是否充分。
- 输出格式是否符合要求。
- 是否有需要补充的材料。

如果某阶段偏了，马上修正，避免最后返工。

### 保存关键产物

长任务容易产生多个结果。建议明确要求保存：

- 阶段计划。
- 数据清洗结果。
- 中间分析表。
- 最终报告。
- 证据清单。

示例：

```text
请把中间分析表保存为 CSV，最终报告保存为 HTML。
```

## 运行中处理

| 状态 | 处理方式 |
| --- | --- |
| 等待补充材料 | 在原会话上传或说明 |
| 等待审批 | 看清楚动作后确认或拒绝 |
| 外部连接失败 | 检查连接状态或改为手动提供材料 |
| 中间结果不对 | 立即修正要求 |
| 任务太大 | 缩小范围或分阶段处理 |

不要重复创建多个相同长任务。优先在原会话继续推进。

## 常见场景

### 周报

```text
请整理本周项目材料，先列出缺失信息，再生成周报草稿，最后保存为 Markdown。
```

### 调研报告

```text
请围绕这个主题做资料整理，先给来源清单和分析框架，确认后再生成完整报告。
```

### 客户跟进

```text
请汇总最近一周客户消息，整理待回复事项、风险客户和建议动作，不要自动发送消息。
```

## 常见问题

### 长任务会自动拆分吗？

不一定。你可以明确要求“先列计划，确认后再继续”，这样更容易控制过程。

### 长任务状态在哪里看？

优先在原来的 Agent会话 中查看。生成的文件通常在工作区文件中查看。

### 中途可以改方向吗？

可以，但要说明哪些结果保留、哪些废弃。例如“保留主题分类，报告结构改成管理层汇报版”。

### 为什么不写命令示例？

因为普通用户主要通过网页提交长任务。底层接口和内部组织方式不作为使用手册主线。
