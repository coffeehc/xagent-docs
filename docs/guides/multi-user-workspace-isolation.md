# 多用户 AI Agent 如何通过虚拟文件系统隔离工作区

当多个人共同使用一套 AI Agent 服务时，只给每位用户建立一个独立目录并不等于完成了文件隔离。真正的隔离还必须控制用户和 Agent 能看到什么、能读取什么、能写入什么，以及哪些系统文件在任何情况下都不能访问。

xAgent 在服务器底层文件系统之上构建了一套内置的虚拟文件系统。Web 页面、Agent 会话和文件工具不会直接访问服务器目录，而是统一通过这套虚拟文件系统获得经过过滤和授权的工作区视图。

## 为什么需要工作区隔离

普通聊天应用主要处理消息，而任务型 AI Agent 还会持续操作文件。例如：

- 用户上传合同、表格、会议记录或源代码。
- Agent 生成 Markdown、CSV、HTML、PDF 或图片。
- 长任务在多个步骤中保存中间材料。
- 工具读取、修改、移动或删除任务文件。

如果只依赖物理目录，文件存在就可能被扫描、读取或误操作。虚拟文件系统则可以把“文件实际存在”和“当前用户或会话是否有权看到它”分开处理。

例如，一个用户的实际工作区中可能已经保存了 1000 个文件，但当前用户界面或某个 Agent 会话只会看到虚拟文件系统允许展示的部分。没有进入当前可见范围的文件，不会因为知道文件名或服务器路径就自动变得可访问。

## xAgent 的隔离方式

xAgent 的工作区隔离由多层规则共同组成，而不是单一目录规则：

1. **用户范围**：每次文件访问都携带当前用户身份，先确定操作属于哪个用户工作区。
2. **虚拟可见范围**：工作区页面和文件工具只返回当前场景允许显示的业务入口与文件，不直接展示物理目录树。
3. **会话访问范围**：每个 Agent 会话都有自己的可读、可写范围，任务只能在虚拟文件系统授予的边界内操作。
4. **代码级硬限制**：系统索引、内部状态、会话元数据、系统私有目录等路径由代码明确保护。即使请求显示隐藏文件或额外配置授权，也不能绕过这些规则。

因此，物理文件是否存在并不直接决定用户或 Agent 是否可见。所有浏览、读取、写入、移动、删除、上传和下载操作，都必须先经过虚拟文件系统的路径归一化、可见性过滤和读写边界检查。

用户通过 Web 查看或下载文件时，接口只使用当前登录用户的身份读取工作区。进入某个会话目录前，系统还会确认该会话属于当前用户。

这意味着两位用户即使都上传名为 `report.xlsx` 的文件，也会分别保存在自己的工作区中，不会因为文件名相同而成为同一个文件。

## 工作区中的业务目录分类

xAgent 不会把用户工作区展示成服务器目录树，而是通过虚拟文件系统整理为几个面向任务的业务入口。当前主要包括：

| 业务目录 | 主要用途 |
| --- | --- |
| 业务空间 | 保存正式业务资料、需要持续复用的文档以及经过确认的结果。适合沉淀不只服务于单次会话的长期材料。 |
| 项目文件 | 保存用户围绕某个项目主动创建和维护的文件，使同一项目的材料、过程文档和结果可以集中管理。 |
| 上传文件 | 保存用户从工作区文件页面上传的本地材料。上传文件只是进入工作区，还需要在任务中说明如何处理。 |
| Agent 会话 | 按会话展示当前用户的任务产物。用户进入某个会话时，主要看到该会话生成的正式结果，而不是全部运行过程文件。 |
| 个人 Skills | 保存当前用户自行创建、导入或持续优化的私有 Skill 文件，其他用户无法在自己的工作区中看到。 |

这些入口属于虚拟工作区的业务分类，不代表用户可以读取其中实际存在的全部文件。xAgent 仍会根据用户身份、当前入口、会话访问范围和代码级保护规则，决定每个文件是否显示以及是否允许读写。

某个业务目录尚未创建或没有可展示内容时，工作区页面可能不会显示对应入口。

## 会话如何限制文件操作

用户隔离解决“这是哪个用户的数据”，虚拟文件系统解决“当前入口能看到什么”，会话访问范围进一步解决“当前任务可以读取和修改什么”。

xAgent 在会话执行时会为工具提供明确的可读和可写范围：

- 文件工具只接受虚拟工作区内的相对路径，不接受服务器绝对路径、网址或向上跳转路径。
- 目录列表是虚拟视图，不等同于服务器上真实存在的全部文件。
- 系统维护目录、内部索引和会话元数据属于硬隐藏目标，不会作为普通任务文件暴露给 Agent。
- 当前会话只能写入明确允许的目录，不能因为文件位于同一个用户工作区就任意修改。
- 额外授权只能扩展普通业务文件的访问范围，不能解锁代码级硬隐藏的系统路径。
- 删除、外发或其他敏感动作还可以继续由审批策略控制。

因此，Agent 操作的是虚拟文件系统提供的授权视图，而不是服务器文件系统本身。

## 上传文件和会话附件

从工作区页面上传的文件会进入当前用户的上传目录，并登记为该用户拥有的文件。遇到同名文件时，系统会生成不同的显示路径，避免覆盖原文件。

从 Agent 会话上传的附件会绑定到当前用户和当前会话。系统会校验附件与会话的归属关系，再允许预览、下载或交给任务处理。附件在会话工作区中的可见文件只是任务使用入口，系统仍保留稳定的文件记录，避免把某个页面路径当作唯一的文件身份。

## 一个简单例子

假设 Alice 和 Bob 共用同一台 xAgent 服务器：

1. Alice 在自己的会话中上传 `customer-list.xlsx`，要求生成客户跟进计划。
2. Bob 也上传一个同名文件，要求统计销售数据。
3. 两个文件分别属于不同用户，并进入各自的会话范围。
4. Alice 的 Agent 无法通过工作区文件工具浏览 Bob 的文件，Bob 也无法打开 Alice 的会话产物。
5. 两个任务都可以生成 `summary.md`，但结果仍保存在各自的用户和会话目录中。
6. 即使 Alice 自己的工作区还存在其他系统文件或未授权文件，当前会话也只能看到虚拟文件系统允许展示的部分。

## 工作区隔离不替代什么

工作区隔离只负责 xAgent 内部的任务文件边界，不应被理解为完整的企业数据权限系统。

- **外部系统权限**：MCP 或连接器访问 CRM、邮箱、数据库等系统时，能读取和修改哪些数据，仍由对应系统的账号授权决定。
- **模型数据边界**：使用外部模型 API 时，发送给模型的数据仍受模型供应商协议约束；需要完全私有化时，应接入本地模型或自有模型网关。
- **部署安全**：服务器账号权限、磁盘加密、备份、HTTPS、防火墙和网络访问控制仍由部署方负责。
- **业务审批**：文件隔离不能判断一次外发或删除是否符合业务规则，高风险动作应结合审批策略治理。

## 部署时应检查什么

1. 为每位实际使用者创建独立账号，不要多人共用同一个 xAgent 用户。
2. 使用反向代理和 HTTPS 保护登录与文件传输。
3. 定期备份 xAgent 运行数据，并验证恢复流程。
4. 为 MCP、连接器和外部账号配置最小必要权限。
5. 用两个测试用户验证同名上传、会话产物、预览和下载不会相互出现。
6. 对删除文件和向外发送数据等动作配置适当的审批策略。
