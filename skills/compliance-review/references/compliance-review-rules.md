# Compliance Review Rules

## Requirement Mapping

For each requirement, capture:

- requirement source, section, owner, authority, and date;
- exact requirement or control objective;
- affected process, system, product, team, data, user, vendor, or region;
- required evidence;
- available evidence;
- status;
- gap, risk, owner, and remediation.

## Status Labels

Use only when evidence supports the label:

- met: evidence shows the requirement is satisfied within scope;
- partially met: evidence supports some but not all required elements;
- not met: evidence shows the requirement is not satisfied;
- not applicable: the requirement is outside the defined scope and the reason is clear;
- unknown: evidence is missing, stale, insufficient, or cannot be verified.

Do not mark a requirement met from policy wording alone. Operating evidence is needed when implementation matters.

## Gap Types

- requirement ambiguity;
- missing owner;
- missing or outdated policy;
- missing procedure;
- missing approval;
- missing training;
- missing implementation;
- missing monitoring;
- missing evidence;
- sample failure;
- retention gap;
- access-control gap;
- disclosure, notice, reporting, or escalation gap;
- vendor, third-party, or subcontractor gap.

## Severity Factors

Consider:

- legal, regulatory, audit, contractual, financial, privacy, security, HR, operational, or reputational impact;
- number of users, customers, employees, vendors, records, transactions, or systems affected;
- sensitive data involved;
- likelihood and repeatability;
- control criticality;
- detectability;
- remediation complexity and timeline;
- existing compensating controls;
- enforcement, audit, or customer sensitivity.

If severity cannot be supported, mark it uncertain.

## Remediation

Good remediation items include:

- action;
- owner;
- due date or priority when known;
- required evidence;
- dependency;
- interim control if needed;
- validation method;
- residual risk after remediation.
