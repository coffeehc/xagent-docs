# Meeting Note Modes

Use this reference when the user asks for lightweight meeting notes, meeting minutes, standup notes, client-call notes, brainstorm notes, project-review notes, or action-only extraction instead of a full recap.

## Quick Notes

Use for rough note cleanup and informal meeting minutes.

Sections:

- meeting details
- summary
- key discussion points
- decisions
- action items
- next steps
- parking lot
- follow-up questions

Keep quick notes compact. Do not add agenda reconciliation or long internal evidence notes unless requested.

## Standup Notes

Use for daily or weekly standups.

Sections:

- each participant's progress
- next work
- blockers
- shared announcements
- follow-up actions

Do not turn standup notes into a formal report unless the user asks.

## Client-Call Notes

Use for client or external partner calls.

Sections:

- meeting objective
- client feedback or requests
- commitments from us
- commitments from client
- follow-up required
- next meeting

Remove internal-only debate from shareable client notes unless the user explicitly asks to preserve private notes.

## Project-Review Notes

Use for milestone or project-review meetings.

Sections:

- progress update
- metrics
- risks and issues
- decisions needed
- action items

Use `weekly-report` instead when the user is reporting recurring status outside a meeting context.

## Brainstorm Notes

Use for idea sessions.

Sections:

- themes
- ideas
- promising options
- objections or constraints
- next experiments
- parking lot

Do not over-rank ideas unless the source material includes criteria or the user asks for prioritization.

## Action-Only Extraction

Use when the user wants only action items.

Required fields:

- action
- owner or `unassigned`
- due date or `not specified`
- source context
- dependency if known
- confidence when inferred

Keep unresolved decisions or unclear asks in a follow-up question section.

## Extraction Signals

Decision signals:

- decided
- agreed
- approved
- confirmed
- final
- go/no-go

Action signals:

- will
- needs to
- follow up
- send
- fix
- review
- confirm
- schedule

Use `unassigned` for ownerless actions and `not specified` for missing dates.
