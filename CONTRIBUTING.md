# Contributing

本文说明 xAgent 文档站的维护规则。

## 文档写作原则

- 使用中文 Markdown。
- 语气客观、清晰、工程化。
- 不写空泛营销文案；项目介绍和企业适用场景可以有产品化表达，但不能承诺未提供的免费定制、托管、集成或支持。
- 使用手册优先，先从功能入口回答用户怎么完成任务。
- 项目介绍集中放在“什么是 xAgent”和首页，不要把每篇文档都写成项目宣传。
- 项目定位统一写成多用户智能体门户，适用于家庭、小团队和小公司快速构建专用智能体。
- 发布定位统一写成免费二进制版本用于产品体验和评估；企业内部系统连接、身份授权、审计、安全策略和专用外部连接通常属于定制集成范围。
- 普通用户页面优先使用中文功能名；只有在解释技术边界、代码/API、配置或固定产品概念时才引入英文术语。
- 涉及具体 UI 操作、配置页面、上传/预览/审批流程时，必要时加入截图；截图应遮盖敏感信息，并放在 `static/img/manual/` 下。
- 不编造未实现功能。
- 不把规划能力写成已实现能力。
- 涉及具体命令、API、配置或行为时，必须回查代码或已有稳定文档。

## 术语一致性

所有文档必须使用 `docs/reference/glossary.md` 中定义的术语。

普通用户页面优先使用中文标题和自然表达；只有在核心概念、技术参考、API、配置或代码说明中，才需要补充英文术语并和术语表保持一致。

新增术语时必须同步：

- `docs/reference/glossary.md`
- 相关概念页

如需修改其他文件，必须说明原因。

## 文档状态说明

`status` 只允许使用：

- `stable`
- `experimental`
- `planned`
- `deprecated`

如果功能尚未稳定，正文必须明确标记：

```md
> 状态：实验性能力，接口可能变化。
```

如果功能尚未实现，正文必须明确标记：

```md
> 状态：规划中。
```

## PR 检查清单

- [ ] 文档使用中文 Markdown。
- [ ] 没有编造未实现功能。
- [ ] 新增术语已同步 `docs/reference/glossary.md`。
- [ ] 链接可以正常访问。
- [ ] `npm run build` 可以通过。
