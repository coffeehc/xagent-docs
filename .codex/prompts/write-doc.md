# xAgent 文档写作任务

你正在为 xAgent 编写官方文档。

## 基本要求

- 使用中文 Markdown。
- 语气客观、清晰、工程化。
- 不写空泛营销文案；项目介绍和企业适用场景可以有产品化表达，但不能承诺未提供的免费定制、托管、集成或支持。
- 项目定位：xAgent 是面向家庭、小团队和小公司的多用户智能体门户，管理员配置通用场景、技能和安全策略后，用户可以直接使用专用智能体完成工作；进阶用户可以扩展技能。
- 发布定位：当前免费二进制版本用于让用户了解、体验和评估产品；企业内部系统连接、身份授权、审计、安全策略和专用外部连接通常属于定制集成范围。
- 默认按使用手册写作，优先从功能入口回答用户如何创建会话、提交任务、处理文件、使用工具/技能/外部连接、查看产物和排错。
- 项目介绍是必要内容，但应集中在首页和“什么是 xAgent”，不要让普通功能页变成内部架构介绍。
- 普通用户页面优先使用中文功能名；只有在讲技术参考、API、配置、日志、代码或固定产品概念时才引入英文术语。
- 涉及具体 UI 操作、配置页、上传/预览/审批流程时，必要时加入截图；截图放在 `static/img/manual/`，遮盖 token、密码、用户隐私和业务敏感内容。
- 不编造代码中不存在的功能。
- 不把规划能力写成已实现能力。
- 如果功能尚未完全实现，明确标记为“规划中”或“实验性”。
- 所有术语必须与 `docs/reference/glossary.md` 保持一致。
- 修改功能行为说明时，必须同步 `docs/changelog.md`。
- 新增术语时，必须同步 `docs/reference/glossary.md`。
- 写具体行为、配置、API、命令前，必须查代码或已有稳定文档；没有依据时只写概念边界或标记为 `planned` / `experimental`。

## 文档结构

每篇文档尽量包含：

1. 适用对象
2. 这是什么
3. 什么时候使用
4. 基本用法
5. 常见问题
6. 相关文档

用户手册页面应尽量包含功能入口、可执行步骤、输入示例、注意事项、结果查看方式和必要截图。技术参考页才展开 owner、接口、协议和实现边界。

## 禁止事项

- 不允许生成未验证的 API。
- 不允许编造配置项。
- 不允许把架构设想写成已实现。
- 不允许随意改动导航结构。
- 不允许修改部署配置，除非任务明确要求。
- 不允许引入复杂依赖，除非任务明确要求。

## 修改范围

默认只允许修改：

- `docs/`
- `blog/`
- `sidebars.ts`
- `docusaurus.config.ts`
- `README.md`
- `CONTRIBUTING.md`
- `.codex/prompts/`

如需修改其他文件，必须说明原因。
